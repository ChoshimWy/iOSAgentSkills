1. 结论先行

核心决策 设计源文件是事实来源，Canonical UI IR 是稳定语义合同，iOS Binding 是组件复用合同，Task Context Compiler 负责为单次任务生成最小 Agent Packet，Implementation Manifest 负责设计到代码的增量追踪，截图 Diff 只承担视觉验收反馈。

高保真 UI 的关键不是向模型提供更多截图或完整 Figma / Sketch JSON，而是把设计工具数据转换成模型可稳定消费的设计语义：页面结构、布局意图、样式 token、组件角色、响应式规则和业务状态。

本方案要求：没有经过 Schema 校验的 Canonical UI IR、明确的设计证据、组件映射、任务级 Agent Packet 和状态矩阵，不得直接进入 SwiftUI/UIKit 高保真实现。

v1.1 的建设重点不再只是“生成 UI IR”，而是形成 Design Evidence → Canonical UI IR → iOS Binding → Task Context Compiler → Agent Packet → Implementation Manifest → Semantic Visual Validation 的完整编译链路。

2. 背景与问题

现有 AI UI 开发通常走“截图 → 代码 → 人工调样式”的路径。截图可以表达最终视觉，却不能可靠表达父子关系、约束意图、组件复用、尺寸变化规则或业务可见条件。另一方面，完整设计工具 JSON 含有大量渲染边界、不可见图层、向量节点、插件数据与实例 override，会显著增加上下文成本。

输入方式优势核心缺陷定位
整屏截图直观、保留视觉目标缺少布局、状态与组件语义视觉证据
完整 Figma / Sketch JSON信息完整噪声高、token 成本大、耦合设计工具模型Parser 输入
人工标注能补业务意图不稳定、难自动校验、易遗漏补充说明
UI IR结构化、可校验、模型无关需要编译器、Schema 和 Registry 建设实现合同

3. 目标、非目标与原则

3.1 目标

  • 建立 Figma / Sketch 到 UIKit / SwiftUI 的稳定设计语义通道。
  • 让 Codex、Claude Code、Gemini CLI 等模型消费同一份基础上下文。
  • 优先复用现有业务组件与 Design System，减少重复实现。
  • 把设计状态、响应式和无障碍约束纳入首轮实现，而不是截图修正阶段。
  • 形成可 lint、可 diff、可回归、可追溯的工程资产。

3.2 非目标

  • 不设计面向 Web/CSS 的通用 UI Schema。
  • 不试图在 v1 自动还原所有复杂矢量、Metal、3D 或视频效果。
  • 不以像素截图替代真实可交互控件。
  • 不新增第二套 iOS 代码实施入口。

3.3 原则

语义优先Constraint Intent 高于绝对 Frame。
真源优先设计文件和目标 Node/Artboard 必须可追溯。
复用优先现有业务组件高于重新创建相似控件。
状态优先页面由业务状态驱动，而不是复刻单张画板。
未知显式化不确定值写入 unknowns，禁止静默猜测。
最窄闭环先锚点状态，再扩展状态矩阵和设备族。

4. 总体架构

Figma / Sketch
      ↓
Design Evidence Snapshot
      ↓
Design Parser → Semantic Normalizer
      ↓
Canonical UI IR ─────────→ design-tokens.json
      ↓
iOS Binding Resolver ────→ ios-component-registry.json
      ↓
Task Context Compiler
      ↓
Agent Packet + reference crop + task instructions
      ↓
Codex / Claude Code
      ↓
UIKit / SwiftUI / mixed-ui
      ↓
Implementation Manifest
      ↓
Semantic Validation + Visual Diff + Review Gate

建议将产物分为五层：

  • Design Evidence：保存可重新解析、可审计的原始设计事实、截图和版本信息。
  • Canonical UI IR：与设计工具无关，描述结构、布局、样式、状态、交互和响应式。
  • iOS Binding：与代码仓有关，描述 UIKit/SwiftUI 类型、源码路径、资产、复用策略和平台限制。
  • Agent Packet：只包含当前任务引用到的节点、token、组件、状态、代码锚点和 reference crop。
  • Implementation Manifest：保存 Design Node、代码符号、PreviewScene、测试和 Diff Region 的双向映射。

这种拆分既能保持 IR 的模型可复用性，也能避免将完整 UI IR 直接塞给模型，并支持设计变更后的增量影响面计算。

6. Canonical UI IR v1.1 数据模型

单一事实源 Canonical UI IR 使用 JSON 保存；YAML 只作为自动生成的可读视图。工具、lint、hash、diff 和版本迁移一律以 JSON 为准。

6.3 Provenance 与可信度

每个可能经过推断的值应记录来源和可信度，避免模型把推断当成设计事实。

corner_radius:
  token: radius.card
  value: 16
  source: design-variable
  confidence: exact

motion:
  duration: null
  source: not-provided
  confidence: unknown

建议支持的 source：design-variable、design-node、component-registry、manual-contract、inferred、not-provided。枚举统一使用 kebab-case。

6.4 版本与迁移

Schema 使用语义版本；新增可选字段可升级 minor，删除、改名或改变语义必须升级 major。Compiler 必须记录 schema_version、parser_version 和迁移记录，避免旧 Agent Packet 静默消费新语义。

7. 布局与 Geometry Intent

约束 UI IR 记录“为什么是这个尺寸”，而不是只记录设计稿在某个 viewport 下的绝对坐标。

布局字段至少应支持：

  • fill、fixed、intrinsic、hug、min、max
  • leading、trailing、top、bottom 和相对节点
  • axis、alignment、distribution、spacing、margin、padding
  • safe area、scroll、sticky、overlay、floating
  • Content Hugging、Compression Resistance、layout priority
layout:
  position: flow
  width:
    mode: fill
  height:
    mode: fixed
    value: 64
  margin:
    horizontal: 24
  top:
    relative_to: mode_selector.bottom
    value: 16

7.1 绝对布局例外

只有装饰图形、Canvas、特殊 Overlay、设计明确要求的自由定位或底层渲染控件可以使用绝对坐标，并必须提供 reason 与适配策略。

8. Design Token 规范

至少建立 Color、Typography、Spacing、Radius、Border、Shadow、Blur、Opacity、Motion、Icon Size 和 Control Height token。所有可复用视觉属性必须先进入 token，再被组件引用。

Token 类型最小字段iOS 映射
Color语义名、RGBA、appearance、来源UIColor / Color
Typographyfamily、size、weight、line height、letter spacingUIFont / Font
Spacing / Radius语义名、pt 值、使用范围Design System 常量
Shadow / Blurx/y/blur/spread/color/materialCALayer / SwiftUI modifier
Motionduration、curve、reduce-motion fallbackUIViewPropertyAnimator / Animation
禁止 不得在页面代码中散落可复用的 HEX、字号、边距、圆角或阴影 magic number。一次性值也必须在页面局部 token 中命名并说明来源。

9. Component Registry 规范

组件匹配顺序固定为：

现有业务组件
→ App Design System 组件
→ Apple 原生组件
→ 新建自定义组件
Control/Slider/Intensity:
  semantic_role: intensity_slider
  reuse_policy: required
  uikit:
    type: SLIntensitySliderView
    source: Modules/LightingUI/Controls/SLIntensitySliderView.swift
  swiftui:
    type: SLIntensitySlider
    source: Modules/LightingUI/SwiftUI/SLIntensitySlider.swift
reuse_policy含义
required必须复用，不得重新实现
preferred优先复用，偏离必须说明原因
native使用 Apple 原生组件
custom-allowed允许新增自定义组件
asset-only必须使用设计资产，不应代码重绘

9.1 自动索引与失效检查

Registry 不应完全依赖人工维护。建议通过 Swift 源码或代码索引采集类型名、模块、源码路径、UIKit/SwiftUI、访问级别、initializer、Configuration、Preview、availability 和最后修改版本，再由人工确认设计映射。

  • 源码路径或 symbol 不存在时标记 stale。
  • required 组件不可访问时阻断 Agent Packet 生成。
  • 跨模块或 availability 不满足时输出 binding conflict，不得静默改为新建组件。

10. Task Context Compiler 与 Agent Packet

v1.1 最重要的优化 AI 不直接消费完整 UI IR。Task Context Compiler 按当前 screen、region、component 或 state 计算引用闭包，只输出实现所需的最小充分上下文。

10.1 裁剪输入

  • 当前任务的目标 node、完整子树、全部祖先和递归约束依赖节点。
  • 目标节点实际引用到的 token、asset 和 Registry binding。
  • 相关业务状态、事件、可见条件和响应式规则。
  • 与目标节点对应的 reference crop、现有代码符号和源码切片。
  • 阻塞性 unknown、推断项和设计冲突。

10.2 Context Budget

任务粒度建议预算超限策略
单组件2K–6K tokens裁掉未引用状态和 token
单区域6K–12K tokens按 component subtree 分片
单页面8K–20K tokens按 region / state 分阶段实现

10.3 Agent Packet 最小合同

11. State、Interaction、Responsive 与 Accessibility

11.1 状态矩阵

一个画板不能代表完整页面。每个页面至少审查 default、selected、pressed、disabled、loading、empty、error、offline、keyboard shown、modal presented、light/dark、超长文案和业务特有状态。

visibility:
  ota_button:
    when: ota.available && !ota.updating
  ota_progress:
    when: ota.updating

11.2 Interaction Graph

静态 scene 之外必须表达 event、from/to state、navigation/presentation、side effect、animation 和失败恢复，防止 AI 只还原单张画板。

transitions:
  - event: tap_ota
    from: ota_available
    to: ota_confirming
  - event: confirm_update
    from: ota_confirming
    to: ota_updating
    side_effect: start_firmware_update
  - event: update_failed
    from: ota_updating
    to: ota_failed

11.3 响应式

  • 记录 compact / regular size class、iPhone / iPad、横竖屏规则。
  • 记录 safe area、键盘、滚动区域、固定区域和最大内容宽度。
  • 尺寸必须表达为关系，例如 parent - 48，而不是固定 345。

11.4 无障碍

  • Dynamic Type、最小点击区域、VoiceOver label/value/hint。
  • 颜色对比度、Reduce Motion、Differentiate Without Color。
  • 无障碍阅读顺序应与 Layout Tree 的语义顺序一致，特殊情况显式覆盖。

12. UIKit / SwiftUI 实现规范

12.1 通用规范

  • 页面必须由 ViewModel / State 驱动，不得把参考截图中的示例状态写死。
  • 实现顺序固定为 tokens → primitive → component → screen → state variants。
  • UI IR 的 node ID、scene ID 应能映射到 PreviewScene、测试或截图命名。
  • 新增组件前必须完成 Registry 查询；命中 required 时禁止重建。
  • 所有 unknown 在编码前明确为阻塞、可接受推断或后续项。

12.2 UIKit

  • 优先使用 NSLayoutAnchor、UILayoutGuide、layoutMarginsGuide、safeAreaLayoutGuide。
  • 正确使用 intrinsicContentSize、Hugging 与 Compression Resistance。
  • 不得把设计稿 x/y/width/height 直接翻译为固定 frame。
  • 状态变化统一由配置方法、content configuration 或状态模型驱动。

12.3 SwiftUI

  • 优先使用 VStack、HStack、Grid、Layout、ViewThatFits。
  • 通过 frame(maxWidth:)、padding、safeAreaInset 表达约束意图。
  • 避免为复刻单一设备截图滥用 GeometryReader 或绝对 offset。
  • Preview 必须注入确定性 fixture，不依赖真实服务。

12.4 Implementation Manifest

实现完成后必须回写设计节点到代码和验证证据的映射，供设计变更影响面分析和最窄回归选择使用。

{
  "designNodeId": "42:18",
  "semanticRole": "intensity_slider",
  "codeSymbol": "SLIntensitySliderView",
  "sourceFile": "FixtureDetailViewController.swift",
  "previewScene": "FixtureDetail/default",
  "validationRegion": "intensity"
}

13. 实施流程与 Checkpoint

Checkpoint必须锁定退出条件
CP0 Intent Lock设计文件、Node/Artboard、版本、设备、状态、UI 框架、兼容边界目标与非目标无歧义
CP1 Anchor Slice根布局、核心组件映射、主要 token、一个 PreviewScene锚点状态方向正确
CP2 Validation Baseline FreezeUI IR、reference、fixture、Diff 环境、ignore mask、阈值验收输入稳定可复现
CP3 Final Gate状态、响应式、无障碍、视觉 Diff、独立 reviewDoD 全部满足

13.1 标准执行顺序

  1. 读取设计真源与目标画板，采集 screenshot、layer tree、variables、styles、symbols。
  2. 生成 Design Evidence Snapshot；Parser 生成 Canonical UI IR，Normalizer 清理设计工具噪声。
  3. 读取现有代码，Registry Resolver 补齐 UIKit / SwiftUI 映射。
  4. 运行 Schema lint，由 Task Context Compiler 生成受预算约束的 Agent Packet。
  5. 实现一个 Anchor Slice 并生成 PreviewScene 截图。
  6. 冻结 baseline 后扩展状态矩阵、响应式与无障碍。
  7. 生成 Implementation Manifest，完成结构、语义、视觉三层验收和独立 code-review。

14. 高保真验收

14.1 三层验收

验收层检查内容不能被替代
结构验收Layout Tree、Registry 复用、token、绝对布局例外不能被截图相似度替代
语义验收状态、显示条件、响应式、多语言、无障碍不能被单一画板替代
视觉验收geometry、spacing、typography、color、asset、z-order不能只靠人工观感

14.2 建议初始阈值

  • 参考 viewport 下关键尺寸和间距偏差不超过 1pt。
  • 字体、字重、行高、文本截断规则精确匹配。
  • 整体截图相似度可从 0.985 起步，但关键组件必须单独设区域门禁。
  • 状态栏时间、动态网络数据、Home Indicator 等忽略项必须进入白名单。
  • baseline 更新必须关联设计版本变化，不得只为消除失败而更新。

14.3 语义辅助 Diff

Diff 失败时应结合 UI IR Geometry、Accessibility Tree 和 Implementation Manifest 定位到具体 node、偏差类型、token 和源码 owner，而不是只返回一张热力图。

node: intensity_label
design_frame: [24, 612, 80, 20]
actual_frame: [24, 614, 80, 19]
difference: { y: 2, height: -1 }
type: typography / geometry
suggested_owner:
  token: typography.bodyMedium
  source: FixtureDetailViewController.swift

15. iOSAgentSkills 集成边界

边界决策 不新增与 ios-feature-implementation 并列的第二个代码实施 Skill。若新增 design-to-ios，其职责应限定为 Design Evidence、UI IR 编译、Context Compiler、Registry 解析和 Agent Packet 交付编排。
模块 / Skill职责输出
ui-ux-design-system设计源解析、Design-to-Code Spec、UI IR 规则设计事实、UI IR source packet
Design Context CompilerEvidence、Adapter、Parser、Normalizer、Schema lint、Registry resolve、上下文裁剪ui-ir.json 与 agent-packet.json
ios-feature-implementationUIKit / SwiftUI / mixed-ui 代码实施生产代码与测试代码
ios-automation截图、UI smoke、无障碍树、设备证据运行截图与自动化证据
ios-verification选择最窄验证并裁决证据验证结论
code-review独立审查组件复用、状态、布局和回归风险阻塞问题与剩余风险

若未来新增或修改 Skill 职责边界，应同步更新 skills/TAXONOMY.md，并按仓库规则执行 Skill Schema lint。

17. 风险与回退

风险影响控制措施回退
设计层级语义差Parser 生成错误组件树Normalizer + 人工确认 Anchor Slice保留 source node 与 unknown
Registry 陈旧AI 复用错误或不存在的组件源码路径校验、owner/version、CI lint标记 unresolved，禁止静默新建
过度追求像素一致破坏响应式和无障碍结构、语义、视觉三层门禁以语义和平台约束优先
Schema 过早复杂化开发和维护成本过高v1 只覆盖最小完整上下文扩展字段放入 namespaced extension
Agent Packet 过度裁剪AI 缺失关键父级、状态或代码上下文引用闭包校验和 context coverage 指标扩大到 region/screen 级 Packet
JSON/YAML 双真源漂移工具与人工看到的语义不一致JSON 为唯一真源，YAML 只允许自动生成删除生成视图并重新导出
模型对 IR 理解不一致跨模型输出波动稳定示例、明确枚举、Schema lint生成模型专用精简 view，不改变真源
Diff 环境不稳定误报和 baseline 漂移固定设备、OS、locale、scale、fixture降级为人工视觉审查并记录缺口

18. Definition of Done

  • 设计文件、页面、Node/Artboard ID 和版本已锁定。
  • Design Evidence Snapshot 包含设计版本、截图 hash 和 Parser version。
  • reference screenshot 与环境信息完整。
  • Canonical ui-ir.json 通过 Schema 校验，YAML 与 JSON 不形成双真源。
  • 不存在未裁决的阻塞性 unknown。
  • 核心组件完成 Registry 映射，required 组件均被复用。
  • Agent Packet 满足 context budget 且引用闭包完整。
  • 布局表达 Constraint Intent，不依赖单一 viewport 的绝对 frame。
  • Design Token、状态矩阵、响应式与无障碍规则完整。
  • 关键交互具有 event、transition、side effect 和失败恢复合同。
  • 关键状态均具有确定性 PreviewScene。
  • Implementation Manifest 可从 Design Node 定位到代码、Preview 和 Diff Region。
  • 结构、语义、视觉三层验收通过。
  • 试点对照报告已记录首轮偏差、组件复用率、token 与人工修正轮次。
  • 独立 reviewer 的 code-review 无阻塞问题。
  • 设计变更后，UI IR、fixture 和 baseline 已同步更新。
