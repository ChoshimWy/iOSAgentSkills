# Understanding Hangs in Your App（摘要）

背景：Apple 关于主线程长时间工作导致 hang 的识别与分析要点。

## 关键概念
- hang 是一次离散交互中的明显停顿，通常大于 `100 ms` 就会被用户察觉。
- hang 几乎总是来自主线程长时间忙碌。
- 主 run loop 会顺序处理 UI 事件、计时器和主队列任务。

## 主线程工作阶段
- 事件先被分发到正确的 view / handler。
- 然后运行你的代码：状态更新、数据获取、UI 变更。
- 最后由 Core Animation 提交到渲染服务器。

## 为什么要关注 main run loop
- 只有主线程可以安全更新 UI。
- run loop 是主线程处理所有 UI 事件的基础。
- 如果 run loop 长时间忙碌，就无法及时处理新事件，用户就会感到卡死。

## 如何诊断 hang
- 观察主 run loop 的忙碌时段；健康状态下它大部分时间应该在休眠。
- Hangs instrument 默认通常会把大于 `250 ms` 的忙碌时段标出来。
- 可以根据需要降低 hang 检测阈值。

## 实用结论
- 主线程工作必须足够短，把重任务从事件处理路径上移走。
- 避免在主队列或主 actor 上挂长时间运行的任务。
- 当你缺少更直接证据时，run loop 的忙碌程度可以作为用户感知卡顿的代理指标。
