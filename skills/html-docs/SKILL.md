# HTML 文档规范

## 角色定位
- 面向“对外分享、评审、归档”的 HTML 文档主格式技能。
- 适用于方案、任务清单、评审稿、分析报告、接口说明、变更说明、运行记录等长文。
- 默认把 HTML 作为主输出；除非用户明确要求，否则不额外生成 Markdown 副本。

## 默认风格
- **以 Notion 风格为主**：浅色、克制、轻边框、轻阴影或无阴影。
- 正文优先，装饰最少；目录、表格、代码块、callout 清晰即可。
- 页面更像“文档阅读”而不是“展示页面”。
- 长文优先单栏阅读，目录可作为页面内导航块；若内容很长可用左侧目录，但不要破坏正文阅读体验。

## 顶部信息规范
- Hero 区顶部元信息必须拆成独立行展示，不要把“创建日期 / 更新日期 / 来源 / 作者 / 仓库 / 版本 / 状态”等拼在同一行。
- 推荐顺序：标题 → 一行主说明 → 独立元信息行（创建日期、更新日期、来源、作者、仓库/版本/状态）→ chips / 标签。
- 若元信息较多，允许继续拆成多行，但每个语义项保持单独一行，避免用 `｜` 过度串联。

## 文件命名规范
- HTML 文档文件名默认不包含日期。
- 文件名优先使用稳定、可复用、便于检索的主题名称，例如 `SidusLinkPro-CUE-ParamSnapshot-Playback-Redesign-Proposal.html`。
- 如需区分不同版本，优先通过文档顶部的创建日期 / 更新日期体现，不通过文件名追加日期。

## 触发判定
- 用户明确要求生成、整理、统一、优化 HTML 文档。
- 用户要求把原有文档导出为 HTML，或后续文档都以 HTML 为主。
- 用户要求统一文档视觉风格、目录、表格样式、代码块样式或阅读体验。

## 使用流程
1. 先判断文档类型：方案 / 任务单 / 接口说明 / 评审稿 / 报告。
2. 按需读取对应 reference：
   - 方案：`references/plan-template.md`
   - 任务单：`references/tasklist-template.md`
   - 接口说明：`references/interface-template.md`
   - 评审 / 分析：`references/review-template.md`
   - 总体风格：`references/template-overview.md`
3. 按统一骨架输出 HTML。
4. 长文必须带目录、锚点和清晰表格。
5. 优先保留信息密度，不为美观牺牲内容。

## 输出标准
- 浅色、克制、正式，默认使用 Notion 风格。
- 标题层级清晰，段落间距充足。
- 表格宽度自适应，长文本自动换行。
- 代码块 / JSON / shell 示例使用等宽字体独立展示。
- 风险、提示、结论、待办使用 callout 区块区分。
- Hero 顶部元信息按独立行输出，避免单行堆叠；默认包含创建日期与更新日期。
- 保留技术名词、路径、API 名称、命令原文。

## 统一骨架
- `hero`
- `toc`
- `section.card`
- `table`
- `pre><code`
- `blockquote` / `callout`
- `footer`

## 约束
- 文件编码 UTF-8，`lang="zh-CN"`。
- 标题、目录、章节锚点必须可跳转。
- 不强行把代码片段翻译成中文伪术语。
- 不在没有要求时同时交付 Markdown 版本。
- 不为了样式牺牲信息密度。
